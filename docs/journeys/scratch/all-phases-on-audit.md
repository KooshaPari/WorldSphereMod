# All-Phases-On Audit

Scope: `VoxelEntities + ProceduralBuildings + CrossedQuadFoliage + MeshWater + HighShadows + SkeletalAnimation + WorldspaceUI + DayNightCycle + PostFX + ParticleEffects`.

## 1) Ordering Hazards

- `Mod.Init()` wires the runtime drivers in a fixed startup order, but the actual frame work is split across multiple Unity `Update`/`LateUpdate` components: `VoxelFrameDriver`, `WorldUIRenderer`, `RuntimeStatsOverlay`, `TimeOfDay`, and `ProceduralSky` ([Mod.cs:38-82](../../../WorldSphereMod/Code/Mod.cs#L38-L82)). The main sink is `VoxelFrameDriver.LateUpdate()`, which flushes meshes, updates water, sun, decals, and post-FX in one pass ([VoxelRender.cs:548-599](../../../WorldSphereMod/Code/Voxel/VoxelRender.cs#L548-L599)).
- That means there is no explicit cross-driver ordering contract in code. If one of the late-frame hooks throws or stalls, everything later in `VoxelFrameDriver.LateUpdate()` for that frame is skipped. The highest-risk coupling points are water lifecycle, sun update, and post-FX application ([VoxelRender.cs:560-599](../../../WorldSphereMod/Code/Voxel/VoxelRender.cs#L560-L599), [WaterRender.cs:16-33](../../../WorldSphereMod/Code/Water/WaterRender.cs#L16-L33), [PostFxController.cs:283-289](../../../WorldSphereMod/Code/Fx/PostFxController.cs#L283-L289)).
- `HighShadows` is effectively snapshot-only after init: `SunDriver.Init()` applies the URP cascade setup once, and runtime phase toggles only patch/unpatch `[Phase]` types; they do not re-run `ShadowCascadeConfig.Apply()` ([SunDriver.cs:18-42](../../../WorldSphereMod/Code/Lighting/SunDriver.cs#L18-L42), [ShadowCascadeConfig.cs:71-128](../../../WorldSphereMod/Code/Lighting/ShadowCascadeConfig.cs#L71-L128), [PhasePatchManager.cs:15-49](../../../WorldSphereMod/Code/PhasePatchManager.cs#L15-L49)).

## 2) Shader Keywords

- I did not find a direct keyword collision between phase shaders. Water uses `_FLAT_WORLD`, sky reserves `_HOSEK_WILKIE`, and voxel rendering uses standard URP instancing/shadow variants only ([WaterGerstner.shader:42-46](../../../WorldSphereMod/Resources/Shaders/WaterGerstner.shader#L42-L46), [ProceduralSky.shader:33-37](../../../WorldSphereMod/Resources/Shaders/ProceduralSky.shader#L33-L37), [VoxelLit.shader:37-47](../../../WorldSphereMod/Resources/Shaders/VoxelLit.shader#L37-L47)).
- The shader risk here is not keyword overlap but mixed fallback materials: water, impostors, UI bars, selection rings, and foliage all have independent shader fallback chains, so a missing runtime shader produces visual inconsistency rather than a shared-keyword crash ([WaterSurface.cs:140-199](../../../WorldSphereMod/Code/Water/WaterSurface.cs#L140-L199), [ImpostorBillboard.cs:30-71](../../../WorldSphereMod/Code/LOD/ImpostorBillboard.cs#L30-L71), [HealthBar.cs:101-107](../../../WorldSphereMod/Code/Worldspace/HealthBar.cs#L101-L107), [SelectionRing.cs:138-146](../../../WorldSphereMod/Code/Worldspace/SelectionRing.cs#L138-L146), [FoliageMaterial.cs:16-30](../../../WorldSphereMod/Code/Foliage/FoliageMaterial.cs#L16-L30)).

## 3) Cumulative Budget

- `MeshInstanceBatcher.Flush()` is the main steady-state cost center. With all render phases on, it is fed by voxel actors, procgen buildings, foliage, and impostors, and it can fall back to per-instance `Graphics.DrawMesh` if instancing breaks ([MeshInstanceBatcher.cs:134-255](../../../WorldSphereMod/Code/Voxel/MeshInstanceBatcher.cs#L134-L255), [VoxelRender.cs:554-598](../../../WorldSphereMod/Code/Voxel/VoxelRender.cs#L554-L598), [BuildingProcRender.cs:19-156](../../../WorldSphereMod/Code/ProcGen/BuildingProcRender.cs#L19-L156), [FoliageTileRender.cs:39-105](../../../WorldSphereMod/Code/Foliage/FoliageTileRender.cs#L39-L105)).
- `SkeletalAnimation` is currently CPU-bound because `RigDriver.SubmitSkinnedActor()` forces `_gpuOK = false`; every humanoid stays on the bind-pose fallback path ([RigDriver.cs:67-129](../../../WorldSphereMod/Code/Rig/RigDriver.cs#L67-L129)).
- `WorldspaceUI` adds another per-visible-actor pass via `WorldUIRenderer.LateUpdate()`, plus `SelectionRing.UpdateAll()`, `DamagePopup.Tick()`, and per-component `NameplateWorld` / `HealthBar` `LateUpdate()` work ([WorldUIRenderer.cs:80-122](../../../WorldSphereMod/Code/Worldspace/WorldUIRenderer.cs#L80-L122), [NameplateWorld.cs:87-110](../../../WorldSphereMod/Code/Worldspace/NameplateWorld.cs#L87-L110), [HealthBar.cs:71-92](../../../WorldSphereMod/Code/Worldspace/HealthBar.cs#L71-L92), [DamagePopup.cs:81-123](../../../WorldSphereMod/Code/Worldspace/DamagePopup.cs#L81-L123), [SelectionRing.cs:53-77](../../../WorldSphereMod/Code/Worldspace/SelectionRing.cs#L53-L77)).
- `ParticleEffects` is event-budgeted, not frame-budgeted, but its pool is capped at 16 systems and silently drops bursts on overflow ([ParticleEffectLibrary.cs:48-56](../../../WorldSphereMod/Code/Fx/ParticleEffectLibrary.cs#L48-L56), [ParticleEffectLibrary.cs:98-120](../../../WorldSphereMod/Code/Fx/ParticleEffectLibrary.cs#L98-L120)).

## 4) Visual Bugs

- I did not find a current same-frame double-draw path for the exact same tree asset from both `ProceduralBuildings` and `CrossedQuadFoliage`. Trees/bushes/palms are auto-routed through `BuildingRulesRegistry.AutoRouteFactory()`, while the tile foliage hook only claims grass/life/road ([BuildingRules.cs:126-146](../../../WorldSphereMod/Code/ProcGen/BuildingRules.cs#L126-L146), [FoliageTileRender.cs:45-52](../../../WorldSphereMod/Code/Foliage/FoliageTileRender.cs#L45-L52)).
- The real overlap bug is stale cross-quad geometry: `CrossedQuadMeshCache` keys only on `(sprite, shape)`, but both call sites pass `swayAmplitude`, so the first build can be reused with the wrong sway or no-sway mesh ([CrossedQuadMeshCache.cs:32-65](../../../WorldSphereMod/Code/Foliage/CrossedQuadMeshCache.cs#L32-L65), [BuildingProcRender.cs:106-119](../../../WorldSphereMod/Code/ProcGen/BuildingProcRender.cs#L106-L119), [FoliageTileRender.cs:83-95](../../../WorldSphereMod/Code/Foliage/FoliageTileRender.cs#L83-L95), [CrossedQuadMesher.cs:17-56](../../../WorldSphereMod/Code/Foliage/CrossedQuadMesher.cs#L17-L56)).
- `HighShadows` stacks with the legacy sprite-shadow path instead of replacing it, so actor and building silhouettes can look too dark/thick when URP shadows and sprite decals both render ([docs/journeys/scratch/high-shadows-audit.md:7-13](../../../docs/journeys/scratch/high-shadows-audit.md#L7-L13), [QuantumSprites.cs:448-553](../../../WorldSphereMod/Code/QuantumSprites.cs#L448-L553), [QuantumSprites.cs:607-673](../../../WorldSphereMod/Code/QuantumSprites.cs#L607-L673)).
