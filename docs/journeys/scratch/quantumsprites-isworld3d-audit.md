# QuantumSprites `IsWorld3D` audit

Scope: the always-on Harmony patches in [`WorldSphereMod/Code/QuantumSprites.cs`](../../../WorldSphereMod/Code/QuantumSprites.cs) at `drawProjectiles`, `drawStatusEffectFor`, `drawBuildings`, and `drawQuantumSprite`. I only checked whether their patch logic avoids 3D work when `Core.IsWorld3D == false`.

## Findings

- `drawProjectiles` is a transpiler, so the patch itself has no runtime branch; it rewrites the target IL to call `Manager.SetProjectile` and `Tools.AddRotation` instead of the original position/rotation flow ([`QuantumSprites.cs:367-379`](../../../WorldSphereMod/Code/QuantumSprites.cs#L367-L379)). The injected `SetProjectile` helper does short-circuit the 3D transform path with `if (Core.IsWorld3D)` and falls back to `localPosition = _last_pos_v3` in 2D ([`QuantumSprites.cs:155-174`](../../../WorldSphereMod/Code/QuantumSprites.cs#L155-L174)). `RotateToCamera` is also gated by `Core.savedSettings.RotateStuffToCamera && Core.IsWorld3D` ([`QuantumSprites.cs:15-21`](../../../WorldSphereMod/Code/QuantumSprites.cs#L15-L21)). Result: no wasted 3D conversion work in 2D from this patch body.

- `drawStatusEffectFor` is also a transpiler, with no runtime branch in the patch itself; it swaps `setScale` for `Manager.SetScaleAndResetRot` and injects `Manager.RotateToCameraAtPos` before `setSharedMat` ([`QuantumSprites.cs:382-393`](../../../WorldSphereMod/Code/QuantumSprites.cs#L382-L393)). `RotateToCameraAtPos` already early-outs on `!Core.IsWorld3D` ([`QuantumSprites.cs:28-33`](../../../WorldSphereMod/Code/QuantumSprites.cs#L28-L33)). `SetScaleAndResetRot` does **not** check `Core.IsWorld3D`; it always calls `setScale` and then `ForceRotation` ([`QuantumSprites.cs:23-27`](../../../WorldSphereMod/Code/QuantumSprites.cs#L23-L27)). That is the only always-on extra work I found here.

- `drawBuildings` is a transpiler with no runtime branch in the patch itself; it replaces `setPosOnly` with `Manager.SetPos` ([`QuantumSprites.cs:395-403`](../../../WorldSphereMod/Code/QuantumSprites.cs#L395-L403)). `SetPos` does short-circuit on `!Core.IsWorld3D`, calling `setPosOnly` and returning before any 3D height conversion (`Tools.To3DTileHeight`) runs ([`QuantumSprites.cs:141-154`](../../../WorldSphereMod/Code/QuantumSprites.cs#L141-L154)). Result: 2D mode is already cheap.

- `drawQuantumSprite` is the only one of the four that has explicit runtime guards in the patch body. Both prefix and postfix return immediately when `!Core.IsWorld3D` ([`QuantumSprites.cs:409-429`](../../../WorldSphereMod/Code/QuantumSprites.cs#L409-L429)). In 3D only, the prefix adjusts `pPos.z` and the postfix rescales `highlight_cursor_zones` ([`QuantumSprites.cs:409-429`](../../../WorldSphereMod/Code/QuantumSprites.cs#L409-L429)). Result: fully short-circuited in 2D.

## Minimal delta

- Only `SetScaleAndResetRot` needs a 2D guard if you want to remove the last always-on extra work. Minimal change: keep `sprite.setScale(pScale)` unconditional, but wrap `sprite.ForceRotation(ref Constants.Zero)` in `if (Core.IsWorld3D)` or return early before that call ([`QuantumSprites.cs:23-27`](../../../WorldSphereMod/Code/QuantumSprites.cs#L23-L27)). No change is needed for `drawProjectiles`, `drawBuildings`, or `drawQuantumSprite` because their 2D paths already skip the 3D-specific work ([`QuantumSprites.cs:155-174`](../../../WorldSphereMod/Code/QuantumSprites.cs#L155-L174), [`QuantumSprites.cs:141-154`](../../../WorldSphereMod/Code/QuantumSprites.cs#L141-L154), [`QuantumSprites.cs:409-429`](../../../WorldSphereMod/Code/QuantumSprites.cs#L409-L429)).
