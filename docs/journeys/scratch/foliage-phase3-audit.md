# Phase 3 Foliage Audit

Scope: `WorldSphereMod/Code/Foliage/FoliageMaterial.cs`, `CrossedQuadMesher.cs`, `CrossedQuadMeshCache.cs`, plus the foliage call sites they feed.

- **(1) Material resolve path: mostly correct fallback, with one latch risk.** `FoliageMaterial.EnsureMaterial()` resolves `Resources.Load<Shader>("Shaders/FoliageWind")` and falls back to `Shader.Find("Sprites/Default")` if the shipped foliage shader is unavailable (`WorldSphereMod/Code/Foliage/FoliageMaterial.cs:16-30`). I did not find any `FoliageLit` lookup in this branch. The only caveat is `_attempted` is sticky: once the first lookup misses, later shader availability will not be re-probed until `Reset()` (`WorldSphereMod/Code/Foliage/FoliageMaterial.cs:13-20,35-40`).

- **(2) Mesh build allocations + cache hits: cache hits are cheap; misses still allocate per build.** `CrossedQuadMeshCache.GetOrBuild()` returns the cached `Mesh` immediately on hit and avoids the mesher allocation path (`WorldSphereMod/Code/Foliage/CrossedQuadMeshCache.cs:32-45`). On a miss it calls `CrossedQuadMesher.Build()` outside the lock, which allocates fresh vertex/uv/color/tris arrays and recalculates bounds/normals (`WorldSphereMod/Code/Foliage/CrossedQuadMesher.cs:17-124`). Eviction also allocates a temporary `List<long>` (`WorldSphereMod/Code/Foliage/CrossedQuadMeshCache.cs:106-127`). More importantly, the cache key is only `sprite.GetInstanceID()` + `shape` and ignores `swayAmplitude` (`WorldSphereMod/Code/Foliage/CrossedQuadMeshCache.cs:35-49`), so a later request with a different amplitude can reuse a stale mesh from the first build. That is a correctness risk, not just a perf detail; `BuildingProcRender` passes `rules.SwayAmplitude` into the cache (`WorldSphereMod/Code/ProcGen/BuildingProcRender.cs:106-116`).

- **(3) Sway shader uniform timing: no obvious one-frame lag.** `WindSwayDriver.LateUpdate()` sets `_WindTime`, `_WindDir`, and `_WindSpeed` every frame (`WorldSphereMod/Code/Foliage/WindSwayDriver.cs:14-24`). The foliage meshes are submitted from the render drivers in the same frame, and the globals are updated before Unity renders. I did not find a CPU-side uniform update delay specific to foliage.

- **(4) Cull-lift latent bug specific to foliage: none found.** The foliage building path lifts `rd.positions[i]` before both culling and TRS construction, and then reuses the lifted position for the final matrix (`WorldSphereMod/Code/ProcGen/BuildingProcRender.cs:42-47,95-129`). The tile path also builds TRS from a lifted 3D tile position (`WorldSphereMod/Code/Foliage/FoliageTileRender.cs:90-95`). I did not find a foliage-specific raw-position-after-cull regression.
