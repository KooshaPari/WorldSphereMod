# DimensionConverter audit

**Date:** 2026-05-20
**Scope:** `WorldSphereMod/Code/DimensionConverter.cs`

## Findings

- **(1) Surfaces patched: 19 explicit method bodies, all wired from `Core.Patch()`.** The startup hook calls `DimensionConverter.Prepare()` once ([`Core.cs:68-70`](../../../WorldSphereMod/Code/Core.cs#L68-L70)), then registers `ConvertPositions` / `ConvertQuantum` against 19 targets: Boulder, Santa, HeatRayEffect, three `QuantumSpriteLibrary` draw methods, four more `QuantumSpriteLibrary` position methods, and eight `GroupSpriteObject.set*` overload variants ([`Core.cs:229-250`](../../../WorldSphereMod/Code/Core.cs#L229-L250)). `TranspilerPosition` rewrites `Transform.position` / `localPosition` gets and sets inside those bodies ([`DimensionConverter.cs:167-200`](../../../WorldSphereMod/Code/DimensionConverter.cs#L167-L200)); `TranspilerQuantum` rewrites set sites only and redirects them to the supplied delegate ([`DimensionConverter.cs:202-223`](../../../WorldSphereMod/Code/DimensionConverter.cs#L202-L223)).

- **(2) No frame loop cost in the converter itself, but the inserted helpers sit on hot paths.** `DimensionConverter` has no `Update`/`LateUpdate`; its work is patch-time. Runtime cost is the extra helper call on each rewritten get/set plus the guard inside `ConvertTo2D` / `ConvertTo3D` (`Core.IsWorld3D`, `v.Is3D()`) before delegating ([`DimensionConverter.cs:42-56`](../../../WorldSphereMod/Code/DimensionConverter.cs#L42-L56)). The expensive part is the same terrain-height lookup already used elsewhere: `Tools.To3DTileHeight` calls `GetTileHeightSmooth` and then `Core.Sphere.SpherePos` ([`Tools.cs:245-269`](../../../WorldSphereMod/Code/Tools.cs#L245-L269), [`Tools.cs:231-233`](../../../WorldSphereMod/Code/Tools.cs#L231-L233)).

- **(3) Wrap-boundary behavior is inherited, not explicit.** The converter delegates x/y mapping to `Tools.To2D` / `Tools.To3D`, which in turn call `Core.Sphere.TilePos` / `SpherePos` ([`Tools.cs:275-281`](../../../WorldSphereMod/Code/Tools.cs#L275-L281), [`Core.cs:412-434`](../../../WorldSphereMod/Code/Core.cs#L412-L434)). That means the converter itself has no seam math. The seam-sensitive part is `GetTileHeightSmooth`, which samples `World.world.GetTile(pos.x + NextPos.x, pos.y + NextPos.y)` for the neighbor tile ([`Tools.cs:245-261`](../../../WorldSphereMod/Code/Tools.cs#L245-L261)). On cylindrical worlds (`CurrentShape == 0`), correctness at `x=0 / width-1` depends on `World.world.GetTile` wrapping cleanly; this file does not make that explicit.

- **(4) Cull-lift formula matches the phase fixes for raw 2D positions.** The three fixes in `dbec1ea`, `1a54da4`, and `974cd81` all lift with `pos = pos.To3DTileHeight(false)` before `Matrix4x4.TRS` ([`VoxelRender.cs`](../../../WorldSphereMod/Code/Voxel/VoxelRender.cs#L379-L390), [`VoxelRender.cs`](../../../WorldSphereMod/Code/Voxel/VoxelRender.cs#L505-L510), [`BuildingProcRender.cs`](../../../WorldSphereMod/Code/ProcGen/BuildingProcRender.cs#L95-L123)). `DimensionConverter.ConvertTo3D` does the same thing for a raw `Vector3` input: it routes through `Tools.To3DTileHeight(v)` after the `Is3D()` guard ([`DimensionConverter.cs:50-56`](../../../WorldSphereMod/Code/DimensionConverter.cs#L50-L56)). Because the `Vector3` overload of `To3DTileHeight` is height-less when `v.z` is still raw 2D data, this agrees with the cull-lift formula used in the fixes. The intentional divergence is `ToQuantumWithHeight`, which preserves `v.z` when the caller already encodes a height payload ([`DimensionConverter.cs:90-99`](../../../WorldSphereMod/Code/DimensionConverter.cs#L90-L99)).

## Bottom Line

No obvious rewrite is needed in `DimensionConverter.cs` for the cull-lift bug class. The main audit risk is the seam-height sampler’s reliance on external wrap behavior; if a future seam bug appears, it is more likely to live in `GetTileHeightSmooth` than in the Harmony transpilers themselves.
