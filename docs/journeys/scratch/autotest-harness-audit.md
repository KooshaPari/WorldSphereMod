# AutoTest Harness Audit

1. The harness is real, but coverage is only partial. `AutoTestDriver` does cycle every `SavedSettings` phase flag, and it loads the newest save directory by mtime rather than hard-coding slot 2. That means the observed `slot=2` in this session is incidental, not a fixed fixture. The render phases that are actually exercised are the ones with real Harmony render hooks: `VoxelRender.ActorVoxelEmit` / `BuildingVoxelEmit`, `BuildingProcRender.ProcMeshEmit`, `FoliageTileRender.Prefix`, `WallTileRender.Prefix`, and the `MeshWater` lifecycle hooks. `SkeletalAnimation` is only reachable inside the `VoxelEntities` path, and `RigDriver` forces `_gpuOK = false`, so AutoTest only covers the CPU bind-pose fallback, not the GPU skinning path. `HighShadows`, `DayNightCycle`, `WorldspaceUI`, `PostFX`, and `ParticleEffects` are not validated by draw-call telemetry alone; some need runtime stimuli (`SelectionHooks`, effect spawns), and `DayNightCycle`/`ProceduralSky` are only created from `Mod.Init` when the flag is already on. `WorldSphereMod/Code/AutoTest.cs:12-24` `WorldSphereMod/Code/AutoTest.cs:86-132` `WorldSphereMod/Code/Voxel/VoxelRender.cs:284-402` `WorldSphereMod/Code/ProcGen/BuildingProcRender.cs:14-157` `WorldSphereMod/Code/Foliage/FoliageTileRender.cs:28-105` `WorldSphereMod/Code/Foliage/WallTileRender.cs:25-68` `WorldSphereMod/Code/Water/WaterRender.cs:35-103` `WorldSphereMod/Code/Rig/RigDriver.cs:67-129` `WorldSphereMod/Code/Worldspace/SelectionHooks.cs:20-73` `WorldSphereMod/Code/Effects.cs:202-214` `WorldSphereMod/Code/Mod.cs:65-82` `WorldSphereMod/Code/Lighting/TimeOfDay.cs:15-22` `WorldSphereMod/Code/Lighting/ProceduralSky.cs:16-22`

2. The output is telemetry, not a pass/fail verdict. The loop only logs `peakDrawCalls`, `peakInstances`, `useFallbackPath`, and `firstActorPos`; there is no assertion, baseline comparison, or nonzero threshold check. A phase can print `0` and still be “green” from the harness’s point of view, which makes the signal useful for manual triage but weak as an automated regression test. `WorldSphereMod/Code/AutoTest.cs:51-73`

3. Gating is present and default-off. `SavedSettings.AutoTest` defaults to `false`, `Mod.OnLoad` turns on `IsAutoTest` from either `WSM3D_AUTOTEST=1` or the saved setting, and `Mod.PostInit` only attaches `AutoTestDriver` when `IsAutoTest` is true. `WorldSphereMod/Code/SavedSettings.cs:35-35` `WorldSphereMod/Code/Mod.cs:30-31` `WorldSphereMod/Code/Mod.cs:88-94`

4. There are phase-transition side effects that can leak across rows. `SetPhase()` mutates `Core.savedSettings` in memory and deliberately skips `SaveSettings()`, so the harness relies on the later `SetPhase(false)` to unwind state. More importantly, `PhasePatchManager` only patches/unpatches Harmony types; it does not call the cleanup methods that actually clear cached state. The codebase has explicit reset paths for voxel materials, instancing fallback, foliage caches, wall meshes, and world-scope lighting cleanup, but those reset methods run on world unload, not on every phase toggle. So a failed or noisy phase can leave `_useFallbackPath`, cached meshes, sky/shadow state, or similar latches alive until the world tears down. `WorldSphereMod/Code/AutoTest.cs:76-84` `WorldSphereMod/Code/PhasePatchManager.cs:15-49` `WorldSphereMod/Code/Voxel/VoxelRender.cs:43-55` `WorldSphereMod/Code/Voxel/MeshInstanceBatcher.cs:344-358` `WorldSphereMod/Code/Foliage/FoliageTileRender.cs:109-112` `WorldSphereMod/Code/Foliage/WallTileRender.cs:130-136` `WorldSphereMod/Code/Lighting/SunDriver.cs:44-53` `WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs:12-31`
