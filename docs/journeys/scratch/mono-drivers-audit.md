# MonoBehaviour drivers audit

Scope: `WorldSphereMod/Code/Voxel/VoxelRender.cs`, `Perf/ProfilerFrameDriver.cs`, `Foliage/WindSwayDriver.cs`, `Lighting/TimeOfDay.cs`, `Lighting/ProceduralSky.cs`, `Water/WaterSurface.cs`, `Worldspace/*`, `ProcGen/DebugSpawnBuildingsDriver.cs`, `AutoTest.cs`, plus unload hooks.

## Findings

1. **Frustum ordering is backwards for the current call graph.** `FrustumCuller.UpdatePlanes()` is only called from `VoxelFrameDriver.LateUpdate()` (`WorldSphereMod/Code/Voxel/VoxelRender.cs:548-558`), but the dependent `IsVisible()` calls happen inside the Harmony postfix emission loops on `ActorManager.precalculateRenderDataParallel` and `BuildingManager.precalculateRenderDataParallel` (`WorldSphereMod/Code/Voxel/VoxelRender.cs:284-401`, `WorldSphereMod/Code/Voxel/VoxelRender.cs:440-527`). That means the culls run before the frame’s plane refresh, so they see stale planes from the previous frame unless some other caller updates them earlier. `FrustumCuller.IsVisible()` also returns `true` when `_planes` is still null, so the first cull pass is permissive by default (`WorldSphereMod/Code/LOD/FrustumCuller.cs:11-25`).

2. **I did not find a driver that belongs in `FixedUpdate`.** The drivers here are render, UI, cache, or bootstrap loops: `TimeOfDay.Update()` advances the day clock and fog, `WindSwayDriver.LateUpdate()` pushes shader globals, `ProfilerFrameDriver.LateUpdate()` flushes profiling, `WaterSurface.LateUpdate()` advances wave time, `WorldUIRenderer/LateUpdate()` and the worldspace widgets are camera-facing UI, and `DebugSpawnBuildingsDriver.Update()` is a one-shot debug bootstrap (`WorldSphereMod/Code/Lighting/TimeOfDay.cs:35-56`, `WorldSphereMod/Code/Foliage/WindSwayDriver.cs:14-25`, `WorldSphereMod/Code/Perf/ProfilerFrameDriver.cs:12-15`, `WorldSphereMod/Code/Water/WaterSurface.cs:130-138`, `WorldSphereMod/Code/Worldspace/WorldUIRenderer.cs:80-122`, `WorldSphereMod/Code/Worldspace/NameplateWorld.cs:87-110`, `WorldSphereMod/Code/Worldspace/HealthBar.cs:71-92`, `WorldSphereMod/Code/ProcGen/DebugSpawnBuildingsDriver.cs:39-77`).

3. **Pause/timeScale sensitivity exists, but I did not find a hard failure.** `RuntimeStatsOverlay` is already pause-safe because it uses `Time.unscaledDeltaTime` (`WorldSphereMod/Code/Worldspace/RuntimeStatsOverlay.cs:83-118`). The other time-driven drivers use scaled time (`Time.deltaTime`, `Time.time`, or `WaitForSeconds`), so pausing will freeze wind, water waves, the fallback day clock, profiler windows, and the auto-test coroutine rather than crash them (`WorldSphereMod/Code/Foliage/WindSwayDriver.cs:14-25`, `WorldSphereMod/Code/Water/WaterSurface.cs:130-138`, `WorldSphereMod/Code/Lighting/TimeOfDay.cs:35-45`, `WorldSphereMod/Code/Perf/ProfilerFrameDriver.cs:14-14`, `WorldSphereMod/Code/AutoTest.cs:26-73`). If the intent is “keep running while paused,” these are the places to switch to unscaled/realtime APIs.

4. **Cleanup coverage is mostly explicit; only one driver uses `OnDisable`.** `DebugSpawnBuildingsDriver` unsubscribes from `MapBox.on_world_loaded` in `OnDisable()` (`WorldSphereMod/Code/ProcGen/DebugSpawnBuildingsDriver.cs:29-37`). Other teardown paths are `OnDestroy()` (`WorldSphereMod/Code/Lighting/TimeOfDay.cs:33-33`, `WorldSphereMod/Code/Lighting/ProceduralSky.cs:33-37`, `WorldSphereMod/Code/Worldspace/RuntimeStatsOverlay.cs:44-50`, `WorldSphereMod/Code/Worldspace/WorldUIRenderer.cs:74-78`) or explicit world-unload hooks (`WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs:12-31`, `WorldSphereMod/Code/Worldspace/WorldUIRenderer.cs:39-63`). I did not see an unhandled event subscription or resource leak in the driver set I checked.
