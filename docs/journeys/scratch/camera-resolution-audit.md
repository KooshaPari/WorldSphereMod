# Camera resolution audit

Scope: `WorldSphereMod/Code/3DCamera.cs` and every `CameraManager.MainCamera` call site in the mod.

1. Call sites are clustered into two groups. Hard dereferences live in `3DCamera.cs:41`, `3DCamera.cs:86`, `3DCamera.cs:131`, and `3DCamera.cs:132`; these assume `MainCamera` already exists and will throw if it does not. Everything else is tolerant or fallback-based: `LOD/LodSelector.cs:46`, `LOD/FrustumCuller.cs:13`, `Voxel/VoxelRender.cs:276`, `Voxel/VoxelRender.cs:545`, `Voxel/VoxelRender.cs:590`, `Voxel/MeshInstanceBatcher.cs:290`, `Voxel/MeshInstanceBatcher.cs:292`, `SunDriver.cs:39`, `SunDriver.cs:66`, `Fx/PostFxController.cs:222`, `Fx/PostFxController.cs:262`, `Worldspace/DamagePopup.cs:85`, `Worldspace/DamagePopup.cs:143`, `Worldspace/NameplateWorld.cs:39`, `Worldspace/NameplateWorld.cs:90`, and `Worldspace/HealthBar.cs:83`.

2. Null behavior is mostly defensive. `LodSelector.Select()` returns `Voxel` when the camera is null; `FrustumCuller.UpdatePlanes()` returns without updating; `VoxelRender.ResolveFlushCamera()` and `MeshInstanceBatcher.ResolveRenderCamera()` fall back to `Camera.main`; `SunDriver.BindMainCamera()` and `SunDriver.Update()` accept null and retry later; `PostFxController`, `DamagePopup`, `NameplateWorld`, and `HealthBar` only assign or orient if the camera exists. The only unconditional assumptions are the `3DCamera` helpers mentioned above.

3. The lifecycle is one-way in current code. `CameraManager.Begin()` creates `MainCamera` once (`3DCamera.cs:107`), and I found no code path that assigns `MainCamera = null` or destroys it explicitly. World mode flips only toggle enabled state and swap `Manager.main_camera` between `OriginalCamera` and `MainCamera` (`3DCamera.cs:87`, `3DCamera.cs:90`, `3DCamera.cs:93`, `3DCamera.cs:96`, `Core.cs:254`, `Core.cs:267`). That means `MainCamera` is expected to be non-null after `Sphere.Prepare()` runs (`Core.cs:436`, `Core.cs:461`); the realistic null windows are startup before `Begin()` and Unity-side teardown/unload, not normal 2D/3D switching mid-frame.

4. The two-camera interaction is the important visibility risk. `MeshInstanceBatcher.ResolveRenderCamera()` prefers the enabled `CameraManager.MainCamera`, otherwise it falls back to `Camera.main` (`MeshInstanceBatcher.cs:288`, `MeshInstanceBatcher.cs:290`, `MeshInstanceBatcher.cs:292`, `MeshInstanceBatcher.cs:295`); `ResolveRenderLayer()` then derives the layer from whichever camera won that race (`MeshInstanceBatcher.cs:298`, `MeshInstanceBatcher.cs:300`, `MeshInstanceBatcher.cs:305`). In a WorldBox scene with a gameplay camera plus `EffectsCamera`, that means the render path is still sensitive to which camera is enabled/tagged as main. The batcher’s current `Camera.allCameras` diagnostic (`MeshInstanceBatcher.cs:320`, `MeshInstanceBatcher.cs:330`) is the right place to confirm the live split. This is the same class of failure as commit `6c7f4b4`: binding draw calls to a specific camera can silently render to the wrong view. The fix in `MeshInstanceBatcher.Flush()` is already to draw with `camera = null` instead of `CameraManager.MainCamera`, which removes that target-camera dependency.
