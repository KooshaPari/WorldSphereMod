# WSM3D Top 5 Integration Risks

Ranked by blast radius and likelihood at `HEAD`:

1. Deferred-destroy queues can be stranded on world unload or after a phase is turned off. `VoxelFrameDriver.LateUpdate()` only drains `ProcGenCache` / `CrossedQuadMeshCache` while `Core.IsWorld3D` is still true and the corresponding phase flag is on, while `WorldUnloadPatch` only calls `Clear()`. If a world finishes after the caches have queued meshes for later destruction, those queued meshes can survive because the drain path is no longer reached. References: [`VoxelRender.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/VoxelRender.cs#L548), [`WorldUnloadPatch.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs#L17), [`ProcGenCache.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/ProcGen/ProcGenCache.cs#L83), [`CrossedQuadMeshCache.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Foliage/CrossedQuadMeshCache.cs#L77).

2. The lighting stack has no wired world-unload teardown. `Mod.Init()` creates `TimeOfDay` and `ProceduralSky`, `SunDriver.Init()` allocates the directional-light root and applies shadow settings, and `SunDriver.Teardown()` exists but is never called from the world-finish chain. Because these are `MonoBehaviour`s attached to the persistent mod object, skybox, fog, and shadow state can bleed across reloads. References: [`Mod.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Mod.cs#L65), [`SunDriver.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/SunDriver.cs#L18), [`TimeOfDay.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/TimeOfDay.cs#L15), [`ProceduralSky.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/ProceduralSky.cs#L16), [`WorldUnloadPatch.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs#L17).

3. `HighShadows` and `DayNightCycle` are write-only at runtime. The UI toggles both settings and persists them, but there is no live apply/destroy path for either flag. `HighShadows` only affects `SunDriver.Init()`, and `DayNightCycle` only gates `TimeOfDay.EnsureCreated()` / `ProceduralSky.EnsureCreated()`, so flipping either one mid-session leaves the current world unchanged until restart. References: [`WorldSphereTab.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/WorldSphereTab.cs#L236), [`SunDriver.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/SunDriver.cs#L18), [`TimeOfDay.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/TimeOfDay.cs#L15), [`ProceduralSky.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Lighting/ProceduralSky.cs#L16).

4. Proc-gen building meshes can poison the cache with fallback cubes. `BuildingMeshGen.Generate()` returns a `UnitCube` for missing or unreadable sprites, and `ProcGenCache.GetOrGenerate()` caches any non-null mesh by `asset.id`. A transient load/readability issue therefore becomes a session-long incorrect mesh for that building id, with no automatic invalidation. References: [`BuildingMeshGen.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/ProcGen/BuildingMeshGen.cs#L10), [`ProcGenCache.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/ProcGen/ProcGenCache.cs#L31).

5. `RigCache` has a dead pending-destroy path. `RigCache.Evict()` enqueues base meshes for deferred destruction, but `RigCache.Clear()` only destroys the currently cached entries and then drops `_pendingDestroy` on the floor. There is a `DrainPendingDestroy()` helper, but it is not wired into the unload path, so evicted rig meshes can leak until process exit. References: [`RigCache.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Rig/RigCache.cs#L140), [`WorldUnloadPatch.cs`](C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs#L24).
