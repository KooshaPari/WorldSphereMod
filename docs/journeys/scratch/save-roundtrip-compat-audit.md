# Save Roundtrip Compatibility Audit

## Verdict
WSM3D does **not** appear to write any mod-specific data into the WorldBox world save JSON. The only explicit serialization path in the mod is `Core.SaveSettings()` / `Core.LoadSettings()`, which reads and writes `mods_config/WorldSphereMod.json`, not the world save. `SavedSettings` is a plain serializable config object, and the world-load/unload hooks only rebuild or clear runtime state. [Core.cs:30-61](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Core.cs#L30) [SavedSettings.cs:5-59](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/SavedSettings.cs#L5) [WorldUnloadPatch.cs:12-31](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs#L12)

## Answers
1. When WSM3D is loaded with `VoxelEntities=false`, I found no code path that changes world-save serialization. The Phase 1 voxel hooks early-return on `!Core.IsWorld3D || !Core.savedSettings.VoxelEntities`, so they only affect runtime render submission, not the saved world JSON. [VoxelRender.cs:284-345](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/VoxelRender.cs#L284) [Core.cs:30-33](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Core.cs#L30)
2. If a world is saved with `VoxelEntities=true` and later opened with the mod uninstalled, I see no evidence that the save file contains WSM3D-only data that would block loading. The mod keeps its 3D state in runtime patches and caches, and those caches are drained on `Core.Sphere.Finish` during normal unload. [WorldUnloadPatch.cs:17-31](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Voxel/WorldUnloadPatch.cs#L17) [General.cs:16-33](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/General.cs#L16)
3. The only persistent data WSM3D writes is its separate settings file under `mods_config`. That file will linger after uninstall unless deleted manually, but it is not part of the world save and does not contaminate the world JSON. [Core.cs:30-33](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/Core.cs#L30)
4. A vanilla save should load cleanly into WSM3D. The mod reconstructs 3D behavior at runtime from the loaded world and `SavedSettings`, and the world-loop patches are gated on `Core.IsWorld3D` / `Core.GeneratingSphere`, so there is no dependency on a pre-existing WSM3D save schema. [General.cs:16-33](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/General.cs#L16) [General.cs:47-76](/C:/Users/koosh/Dev/WorldSphereMod/WorldSphereMod/Code/General.cs#L47)

## Bottom Line
From the code alone: world-save roundtrips look stable, uninstall should not break a saved world, and backward compatibility with vanilla saves is structurally supported. The only state that survives uninstall is the external mod config file.
